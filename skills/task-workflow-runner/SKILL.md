# Task 문서 작업 진행 러너

## 목표
- task tracking 문서를 단일 작업 소스로 사용해 시작/재개 여부를 판단하기
- 각 단계 완료 시 진행 상태를 해당 문서에 기록하기
- 다음 단계로 넘어가기 전에 사용자 피드백을 반드시 받기

## 작업 절차
1. task 문서 찾기
- 사용자가 경로를 지정하면 그 파일을 우선 사용한다.
- 경로가 없으면 `rg --files -g 'TASK.md' -g 'task.md' -g 'WORKSTREAM-TASKS.md'`로 후보를 탐색하기
- 후보가 없으면 사용자에게 파일 경로를 요청하고 응답 전까지 멈추기
- 후보가 여러 개면 목록을 보여주고 대상 파일 선택을 요청하기
- 파일명과 무관하게 `Summary`, `Tasks`, 상태 필드가 있는 task tracking 문서면 사용할 수 있다.

2. 시작/재개 판단하기
- 선택한 task 문서에서 각 항목 상태를 파악하기
- `in_progress`(진행 중) 항목이 있으면 해당 항목부터 재개하기
- 없으면 첫 `not_started`(미시작) 항목부터 시작하기
- `blocked`만 남아 있으면 차단 원인과 재개 조건을 요약해 사용자에게 공유하고 다음 지시를 요청하기
- 모두 완료 상태면 완료 사실을 공유하고 다음 지시를 요청하기
- task ID는 `T1`뿐 아니라 `FW-01-T1` 같은 workstream 기반 ID도 그대로 사용한다.

3. 현재 단계 수행하기
- 단계 시작 전에 목표, 완료 조건, 예상 산출물을 간단히 공유하기
- dependencies가 미충족이면 작업을 시작하지 말고 `blocked` 유지 또는 전환을 검토하기
- 작업 수행을 위한 보조 스크립트를 생성해야 하면 task tracking 문서가 위치한 디렉토리의 하위 디렉토리인 `tools`에 생성하기
- 구현/수정/검증을 수행하기
- 변경 파일, 테스트 결과, 이슈를 정리하기

4. 단계별 진행 기록하기
- 단계 종료 직후 선택한 task 문서를 갱신하기
- 상태 변경, 수행 내용, 검증 결과, 리스크, 다음 행동을 기록하기
- task 항목에 `progress` 필드가 있으면 항상 최신 수행 내용과 검증 결과를 반영하기
- `notes` 필드가 있으면 보조 이력으로 사용할 수 있지만, `next_action`과 상태 갱신을 대신하지는 않는다.
- workstream 구조 문서에서는 기존 bundle/task 계층을 유지한 채 해당 task만 갱신하기
- 로그 섹션이 없으면 `## 진행 로그`를 추가하기

5. 피드백 게이트 적용하기
- 사용자에게 단계 결과를 공유하고 확인/피드백을 요청하기
- 명시적 피드백을 받기 전에는 다음 단계를 시작하지 않기
- 피드백 반영 내용을 task 문서에 추가 기록한 뒤 다음 단계로 진행하기

## 진행 로그 템플릿
```markdown
## 진행 로그
- YYYY-MM-DD HH:MM | 단계: <task_id> | 상태: in_progress -> done
  - 요약: 수행 내용 요약
  - 산출물: 변경 파일/출력/커밋
  - 검증: 테스트 또는 검증 결과
  - 리스크_또는_차단: 남은 리스크 또는 차단 이슈
  - 다음: 다음 단계
  - 사용자_피드백: 승인 대기 / 반영 완료(요약)
```

## 응답 규칙
- 단계 시작 전에는 목표와 완료 조건을 먼저 공유하기
- 단계 종료 후에는 기록 완료 사실과 함께 피드백을 요청하기
- task 문서 형식이 `spec-task-recorder` 또는 `general-task-recorder` 중 어느 쪽이든 기존 필드와 구조를 보존하기
- 경로, 우선순위, 해석이 모호하면 짧게 질문하기
- 사용자 피드백이 기존 계획과 충돌하면 최신 피드백을 우선 반영하기
