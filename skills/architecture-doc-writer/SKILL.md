# 아키텍처 문서 작성

## 작업 흐름

1) 범위 확인
- 대상 파일은 기본적으로 `docs/architecture.md`로 한다. 사용자가 다른 경로를 명시하면 그 경로를 사용한다.
- 코드와 문서를 직접 확인해 사실 기반으로 작성한다. 불확실한 내용은 추측하지 말고 질문한다.

2) 정보 수집
- 리포지토리의 주요 디렉터리와 핵심 모듈을 파악한다.
- 실행 흐름, 데이터 흐름, 외부 의존성(데이터베이스, API, 서드파티 라이브러리)을 확인한다.
- 필요한 경우 Mermaid 다이어그램을 포함할 수 있는지 판단한다.

3) 문서 작성
- 다음 구조(템플릿)로 `docs/architecture.md` 파일을 작성한다.
- 출력은 순수한 Markdown 문서 형식으로 제공한다.
- 문서 최상단 메타 정보는 항상 YAML frontmatter로 작성한다.
- 메타 정보의 표준 키는 `revision`, `updated_at`이다.
- `updated_at`은 타임존을 포함한 ISO 8601 형식으로 기록한다(예: `2026-03-26T14:30:00+09:00`).
- 기존 YAML frontmatter에 다른 필드가 있으면 유지하고, `revision`과 `updated_at`만 추가 또는 갱신한다.
- 본문 제목은 YAML frontmatter 다음 줄에서 `# Architecture`로 시작한다.
- 본문에 표, 목록, 문단 등으로 중복된 메타 정보를 남기지 않는다.
- 필요 시 항목을 추가하되, 아래 필수 범주는 반드시 포함하라.

## 메타 정보 형식

```yaml
---
revision: <git revision>
updated_at: <ISO 8601 datetime with timezone>
---
```

## 필수 포함 내용
1. **문서 정보**: 문서를 작성한 시점의 `revision`, `updated_at`을 문서 최상단 YAML frontmatter로 남긴다.
2. **프로젝트 개요**: 프로젝트의 목적과 핵심 기능을 간단히 설명한다.
3. **전체 아키텍처**: 시스템의 주요 컴포넌트와 상호작용을 설명하고, 아키텍처 다이어그램이 필요한 경우 Markdown 코드 블록에 Mermaid 다이어그램 예시를 포함한다.
4. **디렉토리 구조**: 주요 디렉토리와 파일의 역할을 설명한다.
5. **주요 모듈 설명**: 핵심 모듈별 책임, 입력/출력, 의존성을 정리한다.
6. **데이터 흐름**: 주요 데이터가 어떻게 생성, 변환, 저장, 전달되는지 설명한다.
7. **외부 의존성**: 데이터베이스, API, 라이브러리 등 외부 시스템과의 관계를 기술한다.
8. **확장성 및 유지보수 고려사항**: 앞으로의 확장 전략이나 설계 상 고려사항을 정리한다.

## 출력 형식
- 최종 산출물은 `docs/architecture.md`다.
- 문서는 한국어로 작성한다.
