# SPEC/TASK Evaluator

## 목표
- `SPEC.md`와 `TASK.md`를 기준으로 작업 계획의 품질을 평가한다.
- 수행 가치가 낮은 작업을 제거하고, 누락된 핵심 작업을 보완한다.
- 실행 가능성과 병렬성을 높이도록 작업 범위를 재구성한다.

## 입력 확인
1. `TASK.md` 또는 `task.md`를 찾는다. 둘 다 없으면 평가를 중단하고 누락 사실을 보고한다.
2. `SPEC.md` 또는 `spec.md`를 찾는다. 없으면 코드베이스와 TASK 기준으로 평가하되, SPEC 부재로 인한 한계를 명시한다.
3. 관련 코드 범위를 빠르게 파악한다.
   - 핵심 명령 예시: `rg --files`, `rg "키워드"`, `ls`

## 평가 절차
1. 문서-코드 맥락 정리
- TASK 항목별 목적, 산출물, 의존 관계, 상태를 추출한다.
- SPEC 요구사항(기능/비기능/제약/완료조건)을 추출한다.
- 각 TASK가 실제 코드 변경 단위와 연결되는지 확인한다.

2. ROI 관점 필요성 평가
- 각 TASK를 `impact(영향도)`, `urgency(긴급도)`, `confidence(확신도)`, `effort(노력)`, `risk(위험)`로 평가한다.
- 점수 기준은 [evaluation-rubric.md](references/evaluation-rubric.md)를 사용한다.
- 계산식: `roi_score = (impact * urgency * confidence) / max(effort + risk, 1)`
- 판단:
  - ROI 높음 + SPEC 정합: 유지 또는 우선 수행
  - ROI 낮음 + 정합 약함: 축소, 연기, 제거 후보

3. 잘못된 작업/누락 작업 분석
- 잘못된 작업 식별 기준:
  - SPEC의 어떤 요구사항과도 연결되지 않음
  - 중복 작업이거나 결과가 측정 불가능함
  - 선행 의존성이 누락되어 단독 수행이 불가능함
- 누락 작업 식별 기준:
  - SPEC 완료조건을 만족시키는 TASK가 없음
  - 운영/테스트/마이그레이션/모니터링 등 필수 구현 항목이 비어 있음
- 결과를 "요구사항-작업 매핑" 형태로 제시한다.

4. 범위 적절성 및 분할 가능성 분석
- TASK가 너무 큰 경우, 독립 배포/검증 가능한 단위로 분할한다.
- 분할 원칙:
  - 사용자 가치 또는 기술 경계 기준의 수직 분할 우선
  - 의존성 최소화, 병렬 작업 가능성 최대화
  - 각 단위마다 완료조건(DoD)과 검증 방법 명시
- TASK가 과도하게 잘게 쪼개진 경우 통합 후보를 제시한다.

## 출력 형식
항상 아래 구조로 결과를 작성한다.

```markdown
# 작업 평가 보고서

## 0. 평가 범위
- 대상 문서: SPEC.md, TASK.md
- 코드 범위: (분석한 경로)
- 가정/한계: (누락 문서, 불확실성)

## 1. ROI 필요성 분석
- T1 ... : roi_score=..., 판단=유지/축소/제거, 근거=...
- T2 ...

## 2. 잘못된 작업 및 누락 작업
- 잘못된 작업:
  - ... (근거: SPEC 항목/코드 근거)
- 누락 작업:
  - ... (근거: SPEC 완료조건)
- 요구사항-작업 매핑:
  - R1 -> T1, T3
  - R2 -> (누락)

## 3. 범위/분할 적절성
- 과대 범위 작업:
  - T4 -> T4-1, T4-2, T4-3 (분할 근거)
- 과소 범위 작업(통합 후보):
  - T7 + T8 (통합 근거)

## 4. 권고안
- 즉시 수행:
- 다음 스프린트 이관:
- 제거/보류:
- 우선 확인 질문(있다면):
```

## 유의사항
- 추정 근거 없는 단정은 피하고, 불확실성은 명시한다.
- 파일/코드 근거를 반드시 함께 제시한다.
- 단순 요약보다 "판단 + 근거 + 실행 가능한 조치"를 우선한다.
