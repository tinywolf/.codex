# 아키텍처 문서 작성

## 작업 흐름

1) 범위 확인
- 대상 파일은 기본적으로 `docs/architecture.md`로 한다. 사용자가 다른 경로를 명시하면 그 경로를 사용한다.
- 코드와 문서를 직접 확인해 사실 기반으로 작성한다. 불확실한 내용은 추측하지 말고 질문한다.

2) 정보 수집
- 리포지토리의 주요 디렉터리와 핵심 모듈을 파악한다.
- 실행 흐름, 데이터 흐름, 외부 의존성(데이터베이스, API, 서드파티 라이브러리)을 확인한다.
- 필요한 경우 Mermaid 다이어그램을 포함할 수 있는지 판단한다.

3) 문서 작성
- 아래 “문서 요구사항(프롬프트)”에 따라 `docs/architecture.md`를 작성한다.
- 출력은 순수 Markdown이며 제목은 `# Architecture`로 시작한다.
- 프로젝트 고유 정보는 코드와 문서에서 확인된 내용만 반영한다.

## 문서 요구사항(프롬프트)

아래 요구사항을 그대로 따른다:

```
너는 소프트웨어 아키텍처 문서를 작성하는 기술 문서 작성자 역할을 한다.  

다음 지침에 따라 `docs/architecture.md` 파일을 작성하라:

1. **프로젝트 개요**: 프로젝트의 목적과 핵심 기능을 간단히 설명한다.
2. **전체 아키텍처**: 시스템의 주요 컴포넌트와 상호작용을 설명하고, 아키텍처 다이어그램이 필요한 경우 Markdown 코드 블록에 Mermaid 다이어그램 예시를 포함한다.
3. **디렉토리 구조**: 주요 디렉토리와 파일의 역할을 설명한다.
4. **주요 모듈 설명**: 핵심 모듈별 책임, 입력/출력, 의존성을 정리한다.
5. **데이터 흐름**: 주요 데이터가 어떻게 생성, 변환, 저장, 전달되는지 설명한다.
6. **외부 의존성**: 데이터베이스, API, 라이브러리 등 외부 시스템과의 관계를 기술한다.
7. **확장성 및 유지보수 고려사항**: 앞으로의 확장 전략이나 설계 상 고려사항을 정리한다.

출력은 순수한 Markdown 문서 형식으로 제공하며, 제목은 `# Architecture`로 시작한다.
```

## 출력 형식
- 최종 산출물은 `docs/architecture.md`다.
- 문서는 한국어로 작성한다.
