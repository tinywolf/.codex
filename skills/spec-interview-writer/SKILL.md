# 사용 목적

SPEC.md(또는 사용자 지정 스펙 파일)를 읽어 맥락을 파악한 뒤, 기술·UI/UX·트레이드오프 등 비자명한 심층 질문을 이어가며 정보 공백을 메운다. 사용자가 인터뷰 완료를 확인하면 합의된 내용을 스펙 파일에 작성·갱신한다.

# 기본 흐름

1. **스펙 파일 확인**: 기본은 프로젝트 루트의 `SPEC.md`. 다른 경로가 지시되면 그 파일을 읽는다. 파일이 없거나 접근 불가하면 즉시 알리고 경로를 재확인한다.
2. **기존 맥락 파악**: 파일 핵심 목적, 범위, 현재 확정/미정 항목을 짧게 정리해 공유한다.
3. **질문 설계**: 이미 적힌 내용과 사용자 답변을 피해서, 비자명하고 결정적 영향이 있는 영역(기술/데이터 구조, 실패·경계 조건, 보안/성능, 운영/모니터링, UI 흐름·엣지 케이스, 마이그레이션 등)에 초점을 둔다. 한 번에 3~5개의 질문으로 묶어 짧게 제시한다.
4. **질문 전달**: 준비한 질문을 한 번에 3~5개씩 묶어 사용자에게 제시하고 답변을 받는다. 사용자가 “충분하다/완료”라고 할 때까지 라운드를 반복한다. 각 라운드 이후 새로 드러난 리스크나 결정 사항을 1~2줄로 요약해 피드백한다.
5. **스펙 작성**: 인터뷰 종료 시 다음을 포함해 스펙 파일을 덮어쓴다(기존 핵심이 사라지지 않도록 유지·병합):
   - 목표/비즈니스 배경
   - 범위 내·외
   - 요구사항(기능/비기능)과 우선순위
   - 설계(아키텍처, 데이터 모델, 상태/흐름 다이어그램 요약, API 계약, 에러 처리)
   - UI/UX 시나리오와 엣지 케이스
   - 모니터링/운영/릴리스, 성능/보안 고려사항
   - 리스크·오픈 이슈
6. **검수**: 작성 후 필수 항목 누락 여부, 모순, TODO 표시가 남았는지 점검하고 사용자에게 확인 요청한다.

# 질문 작성 가이드

- “왜/대안/ trade-off/ 실패 시 복구” 관점의 질문을 섞는다.
- 이미 답한 내용이나 파일에 명시된 내용은 반복하지 않는다.
- 모호한 답변이 오면 즉시 좁히는 후속 질문을 추가한다.
- 복수 이해관계자(사용자, 운영자, 개발자, 보안, 데이터)별로 다른 우려를 번갈아 묻는다.
- 숫자·제한치(성능 목표, SLA, 데이터 크기), 단계적 rollout, 실험/feature flag 필요성 등을 확인한다.

# 스펙 작성 팁

- 요약 위주로 간결히 작성하되, 결정된 수치/정책/계약은 명확히 적는다.
- 불확실한 항목은 `TBD`로 표시하고 추가 질문 거리를 남긴다.
- 기존 SPEC.md 구조가 있으면 가급적 보존하며 업데이트한다.

# 산출물

- 최종 스펙이 반영된 `SPEC.md`(또는 지정 파일) 1개.
- 추가 의사결정이 필요한 오픈 이슈 목록(파일 내 섹션 또는 별도 bullet).
